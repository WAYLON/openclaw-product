# 平行 Agent 平台总体架构与交付设计

    ## 第 1 部分：总体架构

    ### 1.1 平行 Agent 架构
    本平台不是一个“万能总脑”，而是一个“专业 Agent 安装与交付平台”。每个业务 Agent 平行存在，各自拥有独立脑子、独立人格、独立记忆命名空间、独立技能绑定、独立文档、独立策略。

    ### 1.2 为什么不要业务总控 Agent
    - 总控 Agent 会让业务边界变模糊，难以交付和培训。
    - 总控 Agent 容易让多个业务知识混入同一人格，难以维护。
    - 渠道实例到业务责任主体不清晰，后期运维和风控困难。
    - 平行业务 Agent 更符合“一个入口实例绑定一个 Agent”的交付原则。

    ### 1.3 各层职责边界
    - `core/`：运行底层，不做业务判断。
    - `channels/`：渠道接入外壳，统一抽象飞书、企微、钉钉、Webhook、Web Chat、API。
    - `agents/<agent>/skills/`：每个 Agent 自带的基础技能、专业技能与第三方接入技能。
    - `agents/`：8 个平行业务 Agent 的独立脑子。
    - `packages/`：最终交付包。
    - `installer/`：安装、打包、升级、下线。

    ### 2.2 每个目录放什么
    - `core/config`：配置加载、配置覆盖、特性开关。
    - `core/memory`：记忆命名空间管理、存储入口。
    - `core/runtime`：Agent 基类、运行时组装。
    - `core/registry`：skill / channel / package 注册表。
    - `core/secrets`：初始化守卫、凭证解析、状态记录。
    - `channels/*`：渠道适配实现。
    - `agents/<agent>/skills/*`：当前 Agent 自带的基础技能、专业技能、第三方接入技能。
    - `agents/<agent>`：每个业务 Agent 的 soul、prompt、knowledge、policy、docs。
    - `packages/<agent>`：给客户交付的资料包目录。
    - `docs/`：平台级教程、讲师资料、模板。
