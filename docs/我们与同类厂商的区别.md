# 我们与同类厂商的区别

这个文档主要给内部售前、技术和交付团队使用，用于统一讲法和产品认知。

它不是写给客户直接阅读的正式宣传页。

为了便于理解，这里统一把市面上的同类方案概括为：

- 代装型
- 教程型
- 提示词型
- 渠道机器人型
- 外壳平台型

无论对方具体叫 QClaw、某某 OpenClaw 工具箱，还是某某小龙虾平台，本质上通常都落在这些类别里。

## 一、我们卖的不是“装上就完”

很多同类方案的核心卖点是：

- 帮你装 OpenClaw
- 帮你接飞书 / QQ
- 帮你切模型
- 给你几个 prompt

这种方案的短板是：

- 装完以后客户不会养
- 没有行业工作流
- 没有清晰的结构设计
- 没有客户专版沉淀
- 后续升级和维护很弱

我们的不同点是：

我们交付的是一整套 **可持续运行、可维护、可扩展的 OpenClaw 产品包**。

## 二、我们卖的是“产品包”，不是零散技巧

很多同类厂商交付的是：

- 零散教程
- 一堆命令
- 若干提示词
- 一个“能聊天”的机器人

我们的交付结构是：

- `core/` 公共底座
- 行业包
- 客户专版
- 安装脚本
- 中文文档
- 安全规范
- 验证脚本

这意味着客户拿到的不是“资料”，而是“产品”。

## 三、我们强调行业化，而不是泛用聊天

很多同类厂商的产品更像：

- 一个能接频道的 AI 助手
- 一个比较会说话的机器人
- 一个平台壳

我们的方向是：

- 股票版
- 教育版
- 后续更多行业版

每个行业包都包含：

- 场景人格
- 工作区规则
- 行业技能
- 行业文档
- 行业验证方式

这才是 OpenClaw 真正容易落地和收费的方向。

## 四、我们支持客户专版，不只卖通用模板

很多方案只能卖“通用版”。

我们的结构天然支持：

- 场景标准版
- 客户定制版

例如：

- `stocks/openclaw-stock-pack`
- `stocks/openclaw-stock-pack-langlang`

也就是说，我们不是交一份通用模板就结束，而是能为客户做出：

- 专属 `SOUL`
- 专属 `AGENTS`
- 专属 `TOOLS`
- 专属技能接入
- 专属验证脚本

这部分是高价值交付的核心。

## 五、我们强调“工具接入”，不是只会聊天

很多同类方案最大的短板是：

- 会说
- 不会做

我们的设计从一开始就强调：

- 工具接入
- 数据接入
- 行业能力接入

例如当前股票方向已经在做：

- `qveris-official`
- 聚宽 / JQData
- `jqdatasdk`
- 后续同花顺

所以我们的定位不是“聊天机器人”，而是“行业工作台”。

## 六、我们强调安全和可运维

很多同类方案只管演示效果，不管后续风险。

我们的仓库已经明确加入：

- `安全交付规范`
- `客户部署清单`
- `默认运行规范`
- `Git` 备份思路
- 验证脚本

这意味着我们更接近真实企业交付，而不是社群里演示一下就结束。

## 七、我们更容易持续升级

很多教程型方案的问题是：

- 交付完成就结束
- 后面升级靠手工
- 难以维护多个客户版本

我们的目录结构从一开始就为升级设计：

- `core/`
- 行业域
- 标准版
- 客户专版

所以后续可以持续做：

- 新版本升级
- 新技能增加
- 新客户专版复制
- 多行业扩展

## 八、我们和同类方案的核心区别

一句话总结：

### 他们更像

- 代装服务
- 教程服务
- 提示词集合
- 机器人接入服务

### 我们更像

- 行业 OpenClaw 产品仓库
- 客户专属 OpenClaw 交付体系
- 可安装、可升级、可维护的 agent 产品方案

## 九、内部售前怎么讲

对外介绍时，不建议说：

- 我们帮你装一个 OpenClaw
- 我们有一些技能

建议说：

- 我们交付的是行业化 OpenClaw 产品包
- 我们支持标准版和客户专版
- 我们能把你的工具和工作流接进 OpenClaw
- 我们交付后还能持续升级和维护
