# 迁移与常见坑说明

这个文档给内部技术安装人员使用。

它不替代安装步骤文档，而是专门记录：

- 从一台机器迁移到另一台机器时最容易漏掉什么
- 安装完成后最常见的失败点是什么
- 应该先查哪里

## 一、迁移时最容易漏掉的东西

### 1. 只装了行业包，没装 core 基础层

常见表现：

- 行业包已经复制过去
- 但很多基础能力并没有准备好
- 安装人员误以为“包已经装完”

正确做法：

1. 先完成 OpenClaw 本体初始化
2. 先执行 `core/scripts/install-core-skills.sh`
3. 再安装行业包

### 2. 只复制了 `skills/`，没处理 `workspace/`

常见表现：

- 技能存在
- 但人格、路由、工具说明没带过去
- 安装后行为和预期不一致

判断方式：

- 看是否使用了 `skills-only`
- 看目标机器是否本来就有自己的工作区

### 3. 复制了 `workspace/`，却没先备份原配置

常见表现：

- 覆盖了目标机器原来的 `SOUL.md`
- 覆盖了原来的 `AGENTS.md`
- 客户原工作方式被直接改掉

正确做法：

完整导入前先备份：

```bash
cp -R ~/.openclaw/workspace ~/.openclaw/workspace.bak.$(date +%Y%m%d_%H%M%S)
cp -R ~/.openclaw/skills ~/.openclaw/skills.bak.$(date +%Y%m%d_%H%M%S)
```

## 二、环境变量相关的常见坑

### 1. `jqdatasdk` 已安装，但聚宽认证失败

典型表现：

- `OK: 已成功导入 jqdatasdk`
- `FAIL: 缺少环境变量 JQ_USER 或 JQ_PASS`

原因：

- 只装了 Python 包
- 没配聚宽账号环境变量

处理：

```bash
export JQ_USER='你的聚宽账号'
export JQ_PASS='你的聚宽密码'
```

长期生效则写入：

- `~/.zshrc`
- `~/.bashrc`

### 2. 环境变量写进了仓库

这是典型错误。

规则：

- 账号密码只能在目标机器本地配置
- 不要写进共享版 `TOOLS.md`
- 不要提交进 Git

### 3. 机器上其实已经配了聚宽账号，但当前 shell 没加载

常见表现：

- `~/.zshrc` 或 `~/.bashrc` 里已经有 `JQ_USER` / `JQ_PASS`
- 但检查脚本仍然报缺失

高频原因：

- 当前终端没有执行 `source ~/.zshrc`
- 检查脚本读取的变量名和现场约定不一致

正确做法：

```bash
source ~/.zshrc
env | grep '^JQ_'
```

先确认当前 shell 已加载，再跑聚宽认证脚本。

### 4. `QVERIS_API_KEY` 没配

表现：

- `qveris-official` 技能能识别
- 但真实调用失败

原因：

- 技能目录在
- 本地密钥不在

### 5. 代理软件装了，但 Node / GUI 进程没有继承代理

常见表现：

- 浏览器能上网
- shell 里 `curl` 也能通
- 但 OpenClaw / Codex OAuth 还是失败

当前机器上的有效样板是：

- 代理软件：`Clash Verge Rev`
- 地址：`http://127.0.0.1:7897`
- 同时写入 shell 和 `launchctl`

也就是说：

只有开代理软件还不够，还要让运行时环境真正继承这些变量。

## 三、技能识别相关的常见坑

### 1. 目录已经复制，但 `openclaw skills check` 里没出现

优先检查：

- 是否复制到了 `~/.openclaw/skills/`
- 目录中是否真的有 `SKILL.md`
- `SKILL.md` 文件名是否正确

### 2. 技能被识别，但不等于真实可执行

例如：

- `qveris-official` 被识别，不等于 Key 已配好
- `jqdata-playbook` 被识别，不等于聚宽已可执行
- `openmaic-official` 被识别，不等于 OpenMAIC 已部署

安装人员不要把“可识别”说成“已打通”。

## 四、股票包迁移时的高频坑

### 1. 通用股票包和 LangLang 包混淆

规则：

- `openclaw-stock-pack`：通用股票包
- `openclaw-stock-pack-langlang`：LangLang 专用包

LangLang 是：

- 通用股票能力
- 再叠加聚宽层

不是只装聚宽。

### 2. 没有 `jqdata-research` 却假装能做历史执行

规则：

- 没有 `jqdata-research`
- 或没有聚宽认证

就只能做研究设计、流程组织、文档约束，不能假装已经跑过真实历史结果。

### 3. `jqdatasdk` 的 warning 被误判成主故障

例如：

- `NotOpenSSLWarning`
- `LibreSSL 2.8.3`

这类 warning 需要关注，但很多时候不是当前失败主因。

优先先看：

- 账号环境变量有没有
- `auth` 能不能过
- `get_price` 能不能跑

## 五、教育包迁移时的高频坑

### 1. 把 OpenMAIC 集成误当成 OpenMAIC 已部署

规则：

- skill 在，不等于服务已跑起来
- 文档在，不等于本地环境已具备

### 2. Node / pnpm 没准备

教育包要优先检查：

- Node.js
- pnpm
- `.env.local`

## 六、媒体包迁移时的高频坑

### 1. `ffmpeg` 不在

常见表现：

- 包装上看起来都有
- 但一到转码或音频处理就失败

### 2. 发送渠道没接入

常见表现：

- 媒体发送包已装
- 但目标渠道根本没配好

这时不能把“有 skill”说成“能发送成功”。

## 七、安装人员推荐排查顺序

一旦装完后不能用，按这个顺序查：

1. OpenClaw 本体是否正常
2. `core` 基础层是否已安装
3. 行业包是否装对目录
4. `openclaw skills check` 是否识别
5. 本地环境变量是否已配
6. 目标依赖是否真的存在
7. 行业专项验证脚本是否通过

## 八、一句话总结

迁移里最常见的误判是：

**“文件过去了”不等于“能力打通了”。**
