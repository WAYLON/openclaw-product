# 产品定位与商业化思路

这个文档主要给内部团队使用，用于统一：

- 我们怎么定义这个仓库
- 我们怎么拆产品层级
- 技术交付时怎么理解标准版和定制版

这个仓库不应只被理解为“OpenClaw 教程仓库”或“技能集合仓库”。

更准确的定位是：

**OpenClaw 行业产品与客户定制交付仓库**

## 一、我们卖的到底是什么

我们卖的不是一个 prompt，不是一个单独 skill，也不是一篇安装教程。

我们卖的是一套已经训练、整理、验证过的 OpenClaw 成果物，交付内容包括：

- `SOUL`
- `AGENTS`
- `TOOLS`
- `USER`
- `MEMORY`
- `skills`
- 行业工作流
- 安装脚本
- 部署文档
- 安全规范

也就是说，交付的是“能运行、能扩展、能维护的 OpenClaw 产品包”。

## 二、为什么这个方向成立

OpenClaw 真正值钱的部分，不在安装，而在“养成”。

价值来自：

- 人格和行为规则
- 长期记忆与工作方式
- 行业化技能
- 本地工具接入
- 客户专属工作流
- 可持续维护能力

所以同样是 OpenClaw：

- 教程型价值：教人装
- 产品型价值：直接交付能用的版本

我们做的是第二种。

## 三、两条商业路径

### 1. 场景标准版

针对某个细分场景，做成可直接安装的标准产品包。

例如：

- 股票研究版
- 教育版
- 小红书运营版
- 设计师版
- 仓储管理版

这类产品的特点是：

- 可重复卖
- 交付成本低
- 易于形成产品矩阵

### 2. 客户定制版

根据客户的实际业务流程和已有工具，做专属定制。

例如：

- 接客户已有 SaaS
- 接客户已有数据源
- 接客户已有内部系统
- 做专属 `SOUL` / `skills` / 工作区规则

这类产品的特点是：

- 客单价更高
- 替代性更低
- 更容易形成长期服务

## 四、当前仓库如何对应这个定位

### 公共底座

- `core/`

作用：

- 安装 OpenClaw 本体
- 初始化工作区
- 提供通用部署方式
- 提供环境变量模板

### 行业产品

- `stocks/`
- `education/`

作用：

- 承载某个行业的标准产品包

### 客户专版

例如：

- `stocks/openclaw-stock-pack-langlang/`

作用：

- 在行业包基础上，增加客户专属工作流、技能、文档和本地验证方式

## 五、内部应如何理解项目卖点

内部不要把项目理解成：

- “我们会装 OpenClaw”
- “我们有一些 skills”

技术和交付团队应统一理解成：

### 卖点 1：私有化部署

不是网页玩具，而是本地 OpenClaw 工作区交付。

### 卖点 2：行业化产品

不是泛用聊天机器人，而是按行业打磨的工作台。

### 卖点 3：客户专版

不是通用模板，而是可按客户定制的专属 OpenClaw。

### 卖点 4：真实工具接入

不是只会聊天，而是能接真实数据、真实系统、真实工作流。

### 卖点 5：可持续维护

不是一次性交付，而是可以继续升级、加技能、调规则、加渠道。

## 六、后续应该怎么继续扩展

建议继续按下面顺序推进：

1. 完善股票标准版
2. 完善 LangLang 客户专版
3. 补教育标准版
4. 补更多行业版
5. 增加企业工具接入能力

## 七、一句话总结

这个仓库的价值，不在于“教人装 OpenClaw”，而在于：

**把 OpenClaw 变成可复制销售的行业产品包，以及可高价交付的客户专属版本。**
