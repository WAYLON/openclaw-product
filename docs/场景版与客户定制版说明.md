# 场景版与客户定制版说明

这个文档是给内部技术与交付团队看的，用于规范目录和版本管理方式。

为了避免项目越做越乱，这个仓库必须明确区分：

1. 场景标准版
2. 客户定制版

## 一、场景标准版是什么

场景标准版是面向某个行业或某类任务的通用产品包。

它的目标是：

- 尽量通用
- 尽量少依赖客户私有环境
- 可快速复制交付

例如：

- `stocks/openclaw-stock-pack/`

这个包就是通用股票研究版。

特点：

- 不绑定某个具体客户
- 不写死客户账号
- 不绑定客户私有工作流

## 二、客户定制版是什么

客户定制版是在场景标准版基础上，为特定客户做的增强交付。

它的目标是：

- 适配客户的真实使用习惯
- 接入客户现有工具和流程
- 形成客户自己的 OpenClaw 工作方式

例如：

- `stocks/openclaw-stock-pack-langlang/`

特点：

- 可以有客户专属的 `SOUL`
- 可以有客户专属的 `AGENTS`
- 可以有客户专属的工具接入
- 可以有客户专属的验证脚本

## 三、为什么一定要分开

如果不分开，项目会出现两个问题：

1. 通用包会越来越脏

客户 A 的配置、客户 B 的规则、客户 C 的脚本都会混在一起。

2. 后续维护会失控

你会搞不清：

- 哪些东西是行业通用能力
- 哪些东西是客户专属逻辑

## 四、命名建议

命名统一按下面规则：

### 场景标准版

```text
<domain>/openclaw-<scenario>-pack
```

例如：

- `stocks/openclaw-stock-pack`
- `education/openclaw-education-pack`

### 客户定制版

```text
<domain>/openclaw-<scenario>-pack-<client>
```

例如：

- `stocks/openclaw-stock-pack-langlang`

## 五、开发原则

### 标准版开发原则

- 保持通用
- 保持轻依赖
- 保持可复制
- 不引入客户专属配置

### 定制版开发原则

- 在标准版上做增量扩展
- 优先复用已有 skill
- 只把客户专属内容放进定制目录
- 不反向污染标准版

## 六、当前项目中的落地方式

### 通用股票版

- `stocks/openclaw-stock-pack`

定位：

- 通用股票研究
- 不带聚宽专属能力

### LangLang 股票定制版

- `stocks/openclaw-stock-pack-langlang`

定位：

- LangLang 专用股票研究版
- 带聚宽 / JQData 能力
- 带客户初始化提示词
- 带最小验证脚本

## 七、后续新增客户时怎么做

新增客户时，技术人员不要修改通用版，应该：

1. 从标准版复制一份
2. 按客户名称建立新目录
3. 加入客户专属规则
4. 加入客户专属脚本
5. 加入客户专属文档

## 八、一句话原则

标准版负责“广泛可卖”，定制版负责“高价可交付”。
