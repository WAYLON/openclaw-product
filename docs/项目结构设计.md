# 项目结构设计

这个仓库按“公共底座 + 行业域 + 产品包 + 文档 + 脚本”五层设计。

## 一、公共底座

目录：

- `core/`

作用：

- 安装 OpenClaw 本体
- 初始化 `~/.openclaw`
- 提供通用环境变量模板
- 提供从 0 部署某个行业包的说明

为什么要单独拆出来：

- 股票和教育都会用到
- 避免业务目录混入公共安装逻辑
- 降低未来扩展成本

## 二、行业域

目录：

- `stocks/`
- `education/`
- `media/`

作用：

- 按行业隔离产品
- 不同行业各自扩展，不互相污染

规则：

- 行业目录只描述该行业的产品
- 不要在行业目录里放本体安装说明

## 三、产品包

示例：

- `stocks/openclaw-stock-pack/`
- `education/openclaw-education-pack/`
- `media/openclaw-av-edit-pack/`
- `media/openclaw-media-delivery-pack/`

每个产品包都应该至少包含：

```text
<pack>/
  README.md
  workspace/
  skills/
```

作用：

- 作为最小交付单位
- 可以直接安装到客户机器

## 四、文档层

目录：

- `docs/`

作用：

- 解释结构设计
- 解释命名规范
- 解释交付规范

## 五、脚本层

目录：

- `scripts/`

作用：

- 安装产品包
- 执行预检查
- 降低手工安装成本

## 结论

这个结构的核心思想是：

- `core/` 解决“OpenClaw 怎么装”
- `stocks/` / `education/` 解决“装什么行业包”
- `<pack>/workspace + skills` 解决“具体装哪些能力”
- `scripts/` 解决“如何标准化安装”
