# 接入飞书

这个文档用于说明如何把 OpenClaw 接入飞书，让客户能够直接在飞书里和小龙虾对话。

## 一、为什么建议接飞书

对大多数客户来说，飞书接入的意义很直接：

- 不用每次打开网页控制台
- 可以在手机上随时下任务
- 更适合企业内部协作
- 更适合把 OpenClaw 做成实际工作台

## 二、接入前准备

请先确认：

- OpenClaw 本体已正常安装
- `~/.openclaw` 已初始化
- 能执行 `openclaw --help`
- 能执行 `openclaw skills check`

## 三、安装飞书工具

执行：

```bash
npx -y @larksuite/openclaw-lark-tools install
```

推荐按飞书官方文章操作：

- [OpenClaw 接入飞书官方说明](https://www.feishu.cn/content/article/7613711414611463386)

## 四、按提示完成飞书机器人接入

安装后，通常会进入扫码或授权流程。

建议：

- 用手机飞书扫码
- 按提示创建机器人
- 完成后台审批

### 实战中更稳的做法

双机器人或第二个机器人场景，不建议把官方安装流程一口气跑到底。

更稳的方式是：

1. 先按官方链接做到“步骤 2”
2. 在安装过程中途停止
3. 去飞书开发者后台找到该机器人的 `App ID` 和 `App Secret`
4. 再把这组 key 明确写进对应 profile 的 `openclaw.json`

也就是：

- 主机器人写到 `~/.openclaw/openclaw.json`
- 第二机器人写到 `~/.openclaw-bot-b/openclaw.json`

这样做的好处是：

- 不容易把第二个机器人误装进默认 profile
- 不容易把主 gateway 和第二 gateway 串线
- 更适合双飞书机器人场景

不要默认假设安装器会自动把第二个机器人落到正确 profile。

如果是给同一台机器接第 2 个飞书机器人，不要复用默认 `~/.openclaw`。

应明确使用独立 profile，例如：

```bash
openclaw --profile bot-b gateway install --port 18799
```

然后把第二个机器人的配置写到：

- `~/.openclaw-bot-b/openclaw.json`

不要写回默认：

- `~/.openclaw/openclaw.json`

否则很容易出现：

- 第二个机器人覆盖第一个机器人
- `ai.openclaw.gateway` 和 `ai.openclaw.bot-b` 指向同一端口
- 主 gateway 被误写成 `bot-b`

### bot-b 常用命令

如果第二个飞书机器人是 `bot-b`，建议固定使用下面这套命令。

#### 1. 查看 bot-b 状态

```bash
openclaw --profile bot-b status
```

#### 2. 查看 bot-b 健康状态

```bash
openclaw --profile bot-b health --json
```

#### 3. 查看 bot-b 的飞书配置

```bash
openclaw --profile bot-b config get channels.feishu
```

#### 4. 修改 bot-b 的飞书配置

例如：

```bash
openclaw --profile bot-b config set channels.feishu.enabled true
openclaw --profile bot-b config set channels.feishu.streaming true
openclaw --profile bot-b config set channels.feishu.connectionMode websocket
```

#### 5. 安装 / 重装 bot-b gateway

```bash
openclaw --profile bot-b gateway install --port 18799
```

#### 6. 重启 bot-b gateway

```bash
launchctl kickstart -k gui/$(id -u)/ai.openclaw.bot-b
```

#### 7. 查看 bot-b 日志

```bash
tail -f /Users/waylon/.openclaw-bot-b/logs/gateway.log
tail -f /Users/waylon/.openclaw-bot-b/logs/gateway.err.log
```

或者：

```bash
openclaw --profile bot-b logs --follow
```

#### 8. 本地直接验证 bot-b 模型链路

```bash
openclaw --profile bot-b tui --session botb-test --message "回复 BOTB_OK" --timeout-ms 20000
```

## 五、接入成功后的验证

验证点包括：

1. OpenClaw 控制台能看到 Feishu 频道
2. 飞书里能搜到机器人
3. 机器人能回复消息

如果是双机器人场景，必须额外验证：

1. 主 gateway 在 `127.0.0.1:18789`
2. 第二个 gateway 在 `127.0.0.1:18799`
3. 主环境可执行：

```bash
openclaw tui --session main-test --message "回复 MAIN_OK" --timeout-ms 20000
```

4. 第二个 profile 可执行：

```bash
openclaw --profile bot-b tui --session botb-test --message "回复 BOTB_OK" --timeout-ms 20000
```

5. 两个命令都必须真正返回文本，而不是只有连接成功

推荐同时检查：

```bash
lsof -nP -iTCP:18789 -sTCP:LISTEN
lsof -nP -iTCP:18799 -sTCP:LISTEN
```

## 六、推荐开启流式响应

为了提升体验，建议开启飞书流式回复：

```bash
openclaw config set channels.feishu.streaming true
```

如果改完配置后未生效，可重启网关：

```bash
openclaw gateway restart
```

如果你是做演示或试用，也可以直接执行仓库里的预设脚本：

```bash
./scripts/apply-demo-open-defaults.sh
```

## 七、常见问题

### 1. 机器人创建成功但无法回复

优先检查：

- 是否完成飞书后台审批
- 是否在 OpenClaw 控制台看到 Feishu 频道
- 网关是否正常运行

如果是双机器人场景，再加查：

- 消息是否真的进入对应 profile 的日志
- 第二个机器人是不是还在指向旧的飞书应用
- `~/.openclaw-bot-b/openclaw.json` 是否真的生效

这次实战里，出现过两类典型现象：

#### 现象 A：飞书里没有输入框

常见原因：

- 打开的不是机器人私聊窗口，而是旧的应用页/残留会话
- 机器人没有按官方流程真正安装成可聊天形态
- 第二个机器人只配了 OpenClaw，没把飞书机器人完整绑定

判断方法：

- 同一个飞书客户端里，别的机器人有输入框，但当前机器人没有
- OpenClaw 侧 `channels.feishu` 已显示 `configured`
- 但飞书会话本身仍然不是正常聊天窗口

处理建议：

1. 先删除旧会话或重新“打开机器人”
2. 确认当前窗口是机器人私聊，不是应用详情页
3. 必要时重新用新的飞书应用创建一个干净机器人

#### 现象 B：有输入框，但发消息不回复

这类要先区分是“模型没回”，还是“消息根本没投递到 OpenClaw”。

本次实战里，最关键的判断方式是：

- 如果日志里没有对应的 Feishu 入站事件
- 那就不是推理失败，而是飞书消息没送进来

优先检查：

1. 飞书后台 `事件与回调` 是否为 `长连接`
2. 是否真的开了消息相关事件订阅
3. 当前聊天窗口是否对应新的 App，而不是旧机器人会话
4. 日志里解析出的 `bot open_id` 是否已经切换到新机器人

#### 现象 C：第二个 gateway 能起，但主 gateway 被串线

本次实战踩到过一次：

- `ai.openclaw.gateway.plist` 被错误写成了 `bot-b` 的内容
- 导致两个 label 都跑到 `18799`

正确状态应该是：

- `ai.openclaw.gateway` -> `18789` -> `~/.openclaw`
- `ai.openclaw.bot-b` -> `18799` -> `~/.openclaw-bot-b`

如果发现串线，先检查：

- `~/Library/LaunchAgents/ai.openclaw.gateway.plist`
- `~/Library/LaunchAgents/ai.openclaw.bot-b.plist`

以及各自的：

- `OPENCLAW_STATE_DIR`
- `OPENCLAW_CONFIG_PATH`
- `OPENCLAW_GATEWAY_PORT`

### 1.1 双飞书机器人推荐最小交付结构

建议固定成：

- 主机器人：`~/.openclaw`
- 第二机器人：`~/.openclaw-bot-b`

并且保证：

- 两份 `openclaw.json` 独立
- 两份 `auth-profiles.json` 独立
- 两个 LaunchAgent 独立
- 两个端口独立

不要把“两个机器人”做成：

- 只有一个 gateway
- 只有一个 state dir
- 只有一份飞书配置反复覆盖

### 2. 流式响应不生效

优先检查：

- `channels.feishu.streaming` 是否已开启
- 是否已经重启网关

### 3. 多媒体消息发送失败

如果后续要让小龙虾发图片、音频、文件，建议：

- 把相关流程沉淀成 skill
- 或补到工作区记忆里

## 八、交付建议

给客户交付时，飞书接入应作为标准可选项，而不是默认强绑。

建议做法：

- 没有飞书需求的客户，不强行配置
- 有飞书需求的客户，再按本文档接入

如果客户明确要“双机器人”：

- 一开始就把 profile、端口、状态目录分开
- 不要先在默认机器人上来回切换 App ID / Secret
- 先确认第一个机器人稳定，再接第二个

## 九、双机器人实战经验总结

这次真实对接 2 个飞书机器人，踩出来的稳定结论是：

1. 第二个机器人必须用独立 profile
2. 第二个 gateway 必须独占端口
3. 不能只改配置文件，还要检查 LaunchAgent 是否真的独立
4. “没有输入框”和“有输入框但不回复”是两类不同问题
5. 如果飞书消息没进入 OpenClaw 日志，就不要继续查模型
6. 先确认消息投递，再确认 agent 执行，再确认回复发送
7. 新旧机器人会话很容易混淆，最好直接用全新的飞书应用和新私聊窗口验证

## 十、一句话总结

飞书接入的价值，不是“多一个聊天入口”，而是把 OpenClaw 变成客户随时可用的工作助手。
