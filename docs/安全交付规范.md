# 安全交付规范

这个文档用于规范 OpenClaw 产品包的对外交付方式，目标是避免因为配置、权限、技能、密钥管理不当而导致安全事故。

如果你在做演示或试用，请同时阅读：

- `默认交付预设.md`

## 一、绝对不要交付的内容

以下内容不得提交到 Git 仓库，也不得直接打包发给客户：

- 真实 API Key
- 聚宽账号密码
- 飞书 / QQ / 机器人 App Secret
- SSH 私钥
- 浏览器 Cookie
- `openclaw.json` 中的敏感认证配置
- `auth-profiles.json` 中的真实授权信息
- 客户机器上的私有路径和敏感目录映射

## 二、必须用占位符替代的内容

以下内容可以保留“模板”，但必须用占位符：

- `env.example`
- `TOOLS.md`
- `USER.md`
- `MEMORY.md`
- 渠道接入说明
- 第三方平台配置说明

## 三、安装前安全建议

给客户交付前，必须明确说明：

1. OpenClaw 不要默认暴露在公网
2. 不要默认开高危工具
3. 不要默认关闭审批（生产环境）
4. 不要默认用管理员权限运行
5. 不要默认安装来路不明的第三方技能

## 四、技能安全要求

安装技能时，优先满足：

- 来源可信
- 最好开源
- 有清晰文档
- 有社区使用记录
- 不要求执行来源不明的脚本

对于客户环境，默认策略应是：

- 先人工审核
- 再安装
- 再做最小验证

## 五、权限原则

OpenClaw 产品包交付时，默认遵循最小权限原则：

- 不需要执行命令时，不开执行工具
- 不需要写文件时，不开写文件工具
- 不需要浏览器时，不开浏览器工具
- 不需要外发消息时，不开消息发送工具

## 六、网络与远程访问

如果客户需要远程使用：

- 优先走 VPN 或 SSH 隧道
- 不建议直接公网暴露
- 必须限制允许访问的来源
- 必须启用身份认证

## 七、客户机器本地配置原则

以下内容必须由客户在本地自行填写：

- 模型提供方配置
- API Key
- 聚宽账号
- 本机脚本路径
- 终端桥接路径
- 频道接入密钥

## 八、关于“全开权限”

如果是为了演示效果，可以采用 `demo/open` 预设，让更多能力默认可见。

但如果是正式交付，不建议把“全开权限”写成唯一默认值。  
更合理的方式是：

- 演示时开放
- 生产时收紧

## 九、交付前检查清单

交付前至少自查：

- 仓库中没有真实密钥
- 安装脚本可运行
- 文档是中文且完整
- `skills check` 能通过
- 客户专用版与通用版边界清楚
- 工作区模板不含私人信息

## 十、事故预防建议

建议客户：

- 用 Git 管理 `~/.openclaw`
- 重要目录先备份
- 高危工具开启审批
- 先在测试环境验证
- 不把实例裸露到公网

## 十一、一句话原则

交付 OpenClaw，不是把“能跑”交出去，而是把“能安全运行、能持续维护、能避免翻车”交出去。
