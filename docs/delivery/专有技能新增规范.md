# 专有技能新增规范

> 这份文档用于约束后续给某个 Agent 新增专有技能时，如何做到增量上线、不破坏已经使用一段时间的客户环境。

## 适用范围

- `stock-agent`
- `education-agent`
- `loan-agent`
- `social-media-agent`
- `news-agent`
- `sales-agent`

`main` 只负责默认入口与技能管理，不作为业务专有技能的主要落点。

## 核心原则

- 只做增量，不做覆盖。
- 不直接改客户线上已经跑着的 `~/.openclaw/openclaw.json` 全量结构。
- 不直接覆盖客户已经修改过的 workspace 文件。
- 不随意修改已有 `agent id`、`memory namespace`、渠道绑定或模型路由。
- 没有真实接通的能力，不写进“已上线能力”。

## 新增专有技能时必须同步更新的地方

以给 `stock-agent` 新增一个专有 skill 为例，至少同步这几处：

1. `agents/<agent>/skills/<skill-name>/`
2. `agents/<agent>/agent.yaml`
3. `agents/<agent>/docs/skills-matrix.md`
4. `agents/<agent>/docs/user-guide.md`
5. `packages/<agent>/config/install-manifest.yaml`
6. 如需培训讲解，再补 `packages/<agent>/docs/trainer-guide.md`

## 每个 Agent 应该怎么更新

### `stock-agent`

- 适合新增：
  - 股票研究增强
  - 数据查询增强
  - 风险总结增强
- 更新方式：
  1. 在 `agents/stock-agent/skills/<skill-name>/` 新增 skill
  2. 在 `agents/stock-agent/agent.yaml` 的专有能力说明中追加
  3. 更新 `agents/stock-agent/docs/skills-matrix.md`
  4. 在 `packages/stock-agent/config/install-manifest.yaml` 写明初始化要求
- 不要做：
  - 不要直接替换现有 `tradingagents-analysis`
  - 不要因为一个股票 skill 改全局模型或全局记忆插件

### `education-agent`

- 适合新增：
  - 教学平台接入
  - 课件处理增强
  - 教学内容生成增强
- 更新方式：
  1. 在 `agents/education-agent/skills/<skill-name>/` 新增 skill
  2. 保留现有 `openmaic`，新增能力按可选启用处理
  3. 更新 `agents/education-agent/docs/skills-matrix.md`
  4. 如果需要 key，明确写清是 LaunchAgent 还是 `openclaw.json`
- 不要做：
  - 不要覆盖现有 `openmaic` 配置
  - 不要把未验证的平台能力写成已上线

### `loan-agent`

- 适合新增：
  - 材料解析
  - 表单辅助
  - 合规提醒增强
- 更新方式：
  1. 先新增 skill 目录
  2. 再补 `skills-matrix.md` 和 `user-guide.md`
  3. 在迁移文档里明确是否读取敏感材料
- 不要做：
  - 不要默认开启高风险自动判断
  - 不要在没有审批的情况下写成正式授信能力

### `social-media-agent`

- 适合新增：
  - 平台浏览增强
  - 内容整理增强
  - 分发辅助增强
- 更新方式：
  1. 优先判断是共享技能还是专有技能
  2. 只有确实只属于社媒场景时，才落到 `agents/social-media-agent/skills/`
  3. 更新 trainer-guide，明确“已接入 / 未接入”
- 不要做：
  - 不要把发布能力默认写成已接管
  - 不要把账号托管写成默认能力

### `news-agent`

- 适合新增：
  - 抓取源接入
  - 聚类增强
  - 简报生成增强
- 更新方式：
  1. 先做文档和模板
  2. 先按可选启用落地
  3. 通过验证后再写进首发能力总览
- 不要做：
  - 不要恢复旧的“已接入但其实没接”的资讯聚类描述

### `sales-agent`

- 适合新增：
  - 线索跟进增强
  - CRM 写入增强
  - 话术与转化分析增强
- 更新方式：
  1. 新增专有 skill
  2. 迁移时只给 `sales-agent` 启用
  3. 验证 CRM 或外部系统写入是否真的打通
- 不要做：
  - 不要因为销售场景的 skill 改动其他 agent 的权限

### `main`

- `main` 不承担业务专有技能
- `main` 只负责：
  - 共享 skill 安装
  - skill 审查
  - skill 分发
  - 路由说明
- 如果某个 skill 是“全体 agent 通用”，应该走共享技能流程，不应伪装成 `main` 的专有技能

## 文档必须写清楚的 4 件事

- 这个 skill 是共享技能还是专有技能
- 是否需要新增 key / token / access code
- 是否默认启用，还是手动启用
- 对老客户属于“新增能力”还是“替换旧能力”

## 默认上线策略

- 新 skill 默认按“可选启用”处理
- 没 key，不启用
- 没依赖，不启用
- 没验证通过，不写进首发能力总览

## 不破坏老用户的推荐做法

### 1. 模板先行

先在 `openclaw-product` 里补模板和文档，不直接改客户线上配置。

### 2. 迁移执行

对已上线客户，走迁移步骤：

1. 拷贝新 skill 到目标 Agent 的 workspace
2. 补齐依赖的 key / 配置
3. 在目标 Agent 增加 skill 条目
4. 重启 gateway
5. 做最小功能验证

### 2.1 现场升级顺序

对已经用了一段时间的客户，推荐永远按下面顺序升级：

1. 先备份客户当前配置
2. 先拷贝 skill 文件，不启用
3. 先补 key / token / access code
4. 先做单 Agent 验证
5. 再把 skill 加到目标 Agent
6. 再重启 gateway
7. 再做真实场景验证

### 2.2 禁止直接做的事

- 禁止“直接覆盖客户整个 `openclaw.json`”
- 禁止“直接替换客户整个 workspace”
- 禁止“先改 agent.yaml 再不做现场验证”
- 禁止“一个 skill 上线顺手改模型、权限、记忆、渠道绑定”

### 3. 回滚简单

回滚时不要回滚整套平台，只回滚该 skill：

1. 从目标 Agent 的 skill 条目里移除
2. 保留 skill 目录但不启用
3. 重启 gateway
4. 确认原有主链路恢复

## 明确禁止的做法

- 禁止把新 skill 直接覆盖到共享技能目录，导致所有 Agent 意外吃到
- 禁止为了一个 skill 修改所有 Agent 的模型、权限或记忆命名空间
- 禁止把未验证通过的 skill 写成“现网已可用”
- 禁止在没有迁移说明的情况下直接替换老 skill

## 发布口径

新增专有技能上线时，对外推荐统一这么写：

- 这是某个 Agent 的新增可选能力
- 需要额外初始化
- 默认不影响已运行客户
- 可按迁移文档启用

## 当前真实案例

- `stock-agent`
  - 已真实接入：`tradingagents-analysis`
- `education-agent`
  - 已真实接入：`openmaic`
- 其他业务 Agent
  - 当前首发先以共享技能为主

## 一句话落地规则

- 共享能力：走共享技能安装与分发
- 专有能力：只落到目标 Agent
- 老客户升级：走迁移，不走覆盖
